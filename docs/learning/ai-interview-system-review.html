真实接口、XML 与调用顺序

src/main/java/com/kun/aiinterview/user/mapper/UserMapper.java

@Mapper
public interface UserMapper {
    User getUserById(@Param("id") Long id);
    User getUserByAccount(@Param("account") String account);
    User getUserByEmail(@Param("email") String email);
    int insertUser(User user);
}

src/main/resources/mapper/UserMapper.xml

XML 的 namespace 指向上述接口，语句 id 与方法名对应；查询使用 #{...}，插入启用生成键回填。

@Mapper 标在接口上，启动扫描时让 MyBatis 创建代理并交给 Spring；否则 AuthService 无法注入实现。@Param 给 XML 中的 #{account} 等占位符稳定命名，属于常见框架约定。调用代理方法后，MyBatis 根据 namespace + 方法名定位语句，预编译绑定值，执行 SQL，再按属性名/枚举类型映射结果。

API/配置正确位置与结果错误风险性质
#{...}Mapper XML；使用预编译参数绑定用字符串替换可能造成 SQL 注入框架固定模式
useGeneratedKeys + keyPropertyinsert 语句；数据库生成 id 后回填 User.id后续拿不到新用户主键框架能力 + 项目选择
resultType=Userselect；将列映射为实体属性列名/枚举不匹配会得到空值或转换错误框架固定模式
4945253 修复前：查询列不完整

       role, status, created_at
FROM user
修复后：实体所需字段补齐

这是实际 Git 修复：登录需要密码密文，安全规则将来还可能读取密码变更时间。漏列不会由 Java 自动补回，最危险的表现是对象存在但关键字段为 null。

测试覆盖的风险

按 id/account/email 查询、插入和生成主键、枚举映射、空结果；目的是验证 XML、数据库列和实体契约真的相符，不只验证接口能编译。

注册：预检查、哈希与数据库唯一约束为何都要有

src/main/java/com/kun/aiinterview/auth/service/AuthService.java

@Transactional
public void register(@Valid RegisterRequest request) {
    if (userMapper.getUserByAccount(request.getAccount()) != null) {
        throw new BusinessException("账号已存在");
    }
    String encoded = passwordEncoder.encode(request.getPassword());
    // 构造 USER + ENABLED 的 User
    try {
        userMapper.insertUser(user);
    } catch (DuplicateKeyException e) {
        throw new BusinessException("账号或邮箱已存在");
    }
}

当前注册链：AuthController.register() 接收 RegisterRequest → MVC 执行 @Valid → AuthService.register() → UserMapper.getUserByAccount/getUserByEmail() → PasswordEncoder.encode() → UserMapper.insertUser() → Controller 返回 Result.success()；校验失败由 MVC 全局异常处理，业务/唯一键错误由 GlobalExceptionHandler 转换。

预检查用于给普通冲突更精确的提示，但两个并发请求可能同时“查不到”；最终必须由数据库唯一索引裁决。DuplicateKeyException 是并发兜底，转成业务异常后由全局处理器输出统一 JSON。@Transactional 让方法内数据库操作处于同一事务，异常时回滚；它由 Spring 代理在外部调用 Service 方法时生效，内部自调用通常绕过代理。

登录：BCrypt 的 encode 与 matches

当前登录链：AuthController.login() → AuthService.login() → 按 account 查 User → PasswordEncoder.matches() → 检查 DISABLED → JwtTokenService.generateAccessToken() → LoginResponse → Result.success()。不存在用户或密码错误提前中断；禁用状态只在密码正确后披露。

src/main/java/com/kun/aiinterview/config/PasswordConfig.java

@Configuration
public class PasswordConfig {
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}

// 注册：保存带随机盐的哈希
passwordEncoder.encode(rawPassword);

// 登录：读取密文并验证，不能重新 encode 后比较字符串
passwordEncoder.matches(rawPassword, encodedPassword);

BCrypt 每次 encode 通常产生不同结果，因为盐被编码进最终字符串；验证时 matches 会从已有密文读取算法参数和盐再计算。@Configuration 告诉 Spring 这是 Bean 定义类，@Bean 把接口类型 PasswordEncoder 放进容器，Service 通过构造器注入，便于替换和测试。

禁止：保存明文、日志打印密码、把密码放进响应/JWT，或用 encode(raw).equals(stored) 验证。

注解/API在当前代码的位置与时机不用/错用后果分类
@RestController、@RequestMappingAuthController；启动时建立 MVC 路由请求无法匹配或返回体不自动序列化框架固定模式
@RequestBody、@ValidController 参数；请求进入时反序列化并校验读不到 JSON 或校验注解不触发框架固定模式
@Service、@ValidatedAuthService；组件扫描并支持方法级校验无法注入；绕过 Controller 调用时参数约束可能不执行框架固定模式
Result.success/errorController/异常处理器；统一业务响应结构前端需处理多种格式项目决策

七、JWT Access Token 代码审计

已实现单元 + 登录集成测试JJWT API 需反复练习

JWT 是服务端签发的“可验证声明”，不是加密容器。Payload 可被解码查看；签名证明内容未被篡改且来自持有密钥的一方，不能证明数据库用户仍启用或角色仍最新。

Header

算法和类型等元数据。

Payload

sub/iss/iat/exp/account/role，可读但受签名保护。

Signature

HS256 使用同一个 SecretKey 签发和验证。

真实配置、生成与验证 API

src/main/java/com/kun/aiinterview/security/jwt/JwtProperties.java · JwtConfiguration.java

JwtProperties 使用 @ConfigurationProperties 将 issuer、Base64 secret 和 access-token-expiration 绑定为类型安全字段，@Validated 在启动时拒绝空值或不合法时长；JwtConfiguration 负责启用这组属性。这样错误配置在启动期暴露，而不是第一次登录时才失败。

src/main/java/com/kun/aiinterview/security/jwt/JwtTokenService.java

// 构造服务时：Base64 文本先解码为字节，再构造 HMAC 密钥
byte[] bytes = Decoders.BASE64.decode(jwtProperties.getSecret());
SecretKey key = Keys.hmacShaKeyFor(bytes);
JwtParser parser = Jwts.parser()
        .verifyWith(key)
        .requireIssuer(jwtProperties.getIssuer())
        .build();

// 登录成功后签发
return Jwts.builder()
        .subject(userId.toString())
        .claim("account", account)
        .claim("role", role.name())
        .issuer(jwtProperties.getIssuer())
        .issuedAt(Date.from(issuedAt))
        .expiration(Date.from(expiration))

        .signWith(key)
        .compact();

// 请求中解析：签名、issuer、exp 等验证成功后取 Claims
Claims claims = parser.parseSignedClaims(token).getPayload();
API调用顺序与结果错用风险
Decoders.BASE64.decode外部配置字符串 → 原始随机密钥字节直接对 Base64 文本调用 getBytes() 得到的是“编码字符”的字节，不是生成 secret 时的原始密钥材料；跨语言/工具验证会不一致，密钥强度判断也针对错误字节
Keys.hmacShaKeyFor字节 → 可供 JJWT 使用的 SecretKey手工构造弱密钥或算法不匹配
Jwts.builder() 链设置声明 → signWith → compact 得到字符串漏 exp 产生过长有效期；放敏感字段会泄露
Jwts.parser() 链verifyWith + requireIssuer → build → parse只解码不校验会信任伪造内容
声明当前来源当前用途权威性
subUser.id过滤器据此重新查询用户标识来源受签名保护，但用户存在性仍查库
issJwtProperties区分签发方，解析时强制匹配验证规则
iat/expInstant + 配置 Duration签发时间和过期时间exp 已验证；iat 尚未用于改密失效
account/role登录时 User携带最少展示性声明role 不用于当前授权，数据库角色才权威

              .requestMatchers("/api/admin/**").hasRole("ADMIN")

              .anyRequest().authenticated())
          .addFilterBefore(jwtAuthenticationFilter,
              UsernamePasswordAuthenticationFilter.class);
        return http.build();
    }
}

@Configuration 和 @Bean 让 Spring 启动时构建过滤链；@EnableWebSecurity 启用 Web Security 配置。HttpSecurity 是链式构建器，http.build() 产出真正执行的 SecurityFilterChain。

配置为什么当前项目这样设不这样做的结果
STATELESS每个请求靠 Bearer Token 恢复身份，不在 HttpSession 保存认证可能出现服务器会话隐式保存，违背 REST 无状态决策
关闭 formLogin/httpBasic已有 JSON 登录接口和 JWT，不需要登录页或 Basic challenge可能得到重定向、HTML 页面或浏览器弹窗
permitAll注册、登录无需先认证用户没有 Token 就无法获得 Token
authenticated其余当前 API 必须有 Authentication接口可能匿名访问
hasRole("ADMIN")验证管理员 URL；它会为参数补 ROLE_ 并检查 ROLE_ADMIN。若用 hasAuthority，则应传完整字符串 ROLE_ADMIN错误混用前缀会让合法管理员恒定 403
addFilterBefore授权判断前先恢复 Context；锚点是用户名密码过滤器授权执行时仍是匿名用户
JwtAuthenticationFilter：无 Token、坏 Token、好 Token 三条路径
没有 Authorization

resolveBearerToken 返回 null，不创建身份，继续 filterChain.doFilter。因为“是否允许匿名”属于授权规则：公开端点可通过，受保护端点稍后进入 401。

Header/Token 无效

空 Header、非 Bearer、空 Token、签名/过期/issuer/subject 异常转换为 AuthenticationException；清空 Context，EntryPoint 写 401，然后 return，不能继续链路。

Token 有效

从 sub 取 userId → 查数据库 → 检查 ENABLED → 将最新角色映射为 authority → 创建 Authentication → 设置 Context → 继续授权。

src/main/java/com/kun/aiinterview/security/filter/JwtAuthenticationFilter.java

Authentication authentication =
    UsernamePasswordAuthenticationToken.authenticated(
        new AuthenticatedUser(id, account, username),
        null,
        List.of(new SimpleGrantedAuthority("ROLE_" + role.name()))
    );
SecurityContext context = SecurityContextHolder.createEmptyContext();
context.setAuthentication(authentication);
SecurityContextHolder.setContext(context);

当前项目的精确调用链：SecurityConfiguration.securityFilterChain() → JwtAuthenticationFilter.shouldNotFilter() / doFilterInternal() → resolveBearerToken() → authenticate() → JwtTokenService.parseAndValidate() → Claims.getSubject() → UserMapper.getUserById() → loadEnabledUser() → createAuthentication() / mapAuthority() → SecurityContextHolder.setContext() → 后续授权过滤器 → Controller。任一步认证异常都由 EntryPoint 输出 401 并提前结束。

Authentication 是当前身份快照：principal 是谁，credentials 是凭据（请求恢复身份后不再保存密码，所以为 null），authorities 能做什么，authenticated 表示是否已认证。SecurityContext 容纳 Authentication，SecurityContextHolder 让同一请求线程上的后续安全组件和 Controller 获取 Context。

shouldNotFilter、permitAll 与重复注册

src/main/java/com/kun/aiinterview/security/filter/JwtAuthenticationFilter.java

OncePerRequestFilter 提供固定扩展点：shouldNotFilter 决定本过滤器是否跳过，doFilterInternal 处理一次请求。登录/注册使用 shouldNotFilter 跳过，是项目安全决策：即使客户端带了过期旧 Token，也不应阻止重新登录或注册。

容易混淆：permitAll 是授权阶段允许访问，不会自动阻止自定义 JWT Filter 执行。因此只写 permitAll 而不跳过滤器，旧坏 Token 仍可能在登录到达 Controller 前触发 401。

src/main/java/com/kun/aiinterview/security/config/SecurityConfiguration.java

JwtAuthenticationFilter 同时是 @Component 和 Servlet Filter。Spring Boot 可能把 Filter Bean 自动注册进普通 Servlet 链；安全配置又把它加入 Security 链。FilterRegistrationBean.setEnabled(false) 禁用普通自动注册，确保只按 Security 链顺序执行一次。

Spring Security 固定模式本项目自己的安全决策
提供 SecurityFilterChain Bean；过滤器扩展 OncePerRequestFilter；构造 Authentication；写入 SecurityContextHolder；配置 EntryPoint/DeniedHandler无状态 JWT；登录/注册跳过旧 Token；sub 只用作查库键；数据库最新 status/role 权威；USER/ADMIN 映射 ROLE_ 前缀；当前 /api/admin/** 限 ADMIN
排错顺序：先看请求是否进入过滤链；再看 shouldNotFilter；检查 Header scheme 和 Token parser；检查 sub 与数据库用户；检查 role 是否映射为 ROLE_*；确认 Authentication 已放入 Holder；最后看 URL 匹配顺序和 401/403 处理器。

为什么 GlobalExceptionHandler 不能完全处理过滤器异常？

@RestControllerAdvice 属于 Spring MVC 异常解析体系，通常在请求已经进入 DispatcherServlet、Controller 调用链后生效。Security Filter 在 MVC 之前执行；过滤器里抛出的认证异常不保证经过 ControllerAdvice。因此当前项目在过滤器中捕获认证异常并直接调用 EntryPoint，授权异常则由 Security 的异常处理机制交给对应 handler。

十三、讲义：JWT 心智模型

问题正确理解
JWT 是否加密？当前 JWS 不是。Payload 可读，不能放密码、密文、邮箱等不必要数据。
签名验证成功是否等于用户可用？不等于。它只验证 Token；当前项目再查数据库 status/role。
为什么 HS256 两边同一密钥？HMAC 是对称签名，签发和验证共享 SecretKey；泄露即失去信任边界。
Base64 secret 是什么？密钥字节的文本编码，不是“再加密”。应用启动时解码并检查强度。
为什么只做 Access Token？当前阶段严格控制范围；刷新、黑名单、轮换会引入新的状态和安全规则。
退出登录怎么办？当前未实现服务端即时撤销；Token 到期前仍可能有效，除非用户状态变化被查库拒绝。
一口气解释：用户正确登录后，服务端用外部密钥签发短期 Access Token。客户端每次带 Bearer Token；服务端校验签名、issuer 和 exp，再用 sub 查数据库。Token 负责携带受签名保护的标识，数据库负责当前状态和角色，SecurityContext 负责本次请求中的框架身份。

Filter 与 HandlerInterceptor 为什么不能互换

Servlet Filter 位于 DispatcherServlet 之前，能覆盖所有进入 Web 应用的请求，也是 Spring Security 的原生执行模型。HandlerInterceptor 属于 Spring MVC，只围绕已经匹配到的 Controller。JWT 身份必须在 Security 授权过滤器执行前建立，所以放在 Security Filter Chain，而不是 MVC 拦截器。

手工复现的最小步骤
  1. 声明 SecurityFilterChain Bean 和 URL 规则。
  2. 继承 OncePerRequestFilter，实现跳过规则和 doFilterInternal。
  3. 读取 Bearer，调用专门 Token 服务验证。
  4. 只从 sub 得到 id，查数据库最新用户。
  5. 将数据库 role 映射为 ROLE_ authority。
  6. 创建 authenticated Authentication，放入新 Context，再写入 Holder。
  7. 配置 401/403 JSON handler，并阻止 Filter 重复注册。
  8. 分别测试无 Token、坏 Token、好 Token、禁用用户、角色变更和权限不足。

当天代码事实：建立 SecurityFilterChain、OncePerRequestFilter、Authentication、SecurityContext、数据库状态/角色校验、401/403 JSON handler、重复 Filter 注册禁用和集成测试。
